# Postmortem: GitHub Service Outage — October 21-22, 2018

**Date:** October 21-22, 2018  
**Duration:** ~24 hours  
**Severity:** Critical (P1)  
**Services Impacted:** Issues, Pull Requests, Webhooks, Pages, API (write operations)

---

## Executive Summary

22 октября 2018 года GitHub пережил серьезный сбой, который привел к отказу большинства операций записи на платформе в течение 24 часов. В ходе плановой замены вышедшего из строя сетевого оборудования произошло нарушение связи между сетевыми хабами США, что привело к рассогласованию данных в кластере MySQL и длительной процедуре восстановления.

## Root Cause

1. **Физический сбой:** Отказ 100G оптического модуля в сетевом оборудовании Восточного побережья США.

2. **Сетевая нестабильность:** После замены модуля сетевое соединение между дата-центрами было восстановлено, но оставалось нестабильным в течение 43 секунд. Это привело к частичной потере пакетов и "раздвоению мозга" (split-brain) в кластере MySQL.

3. **Рассогласование данных:** Из-за кратковременной потери связи реплики MySQL в западном дата-центре "отстали" от мастера на восточном побережье. При восстановлении связи началась репликация с расхождением данных.

4. **Защитное отключение:** Чтобы предотвратить необратимую потерю данных, GitHub немедленно остановил запись в основную базу данных.

---

## Impact

- **24 часа** невозможности записи: нельзя было создавать Issues, Pull Requests, пушить код, обновлять Wikis, Pages.
- **Операции чтения** (клонирование, просмотр кода) работали с задержками из-за нагрузки на кэш.
- Миллионы разработчиков по всему миру не могли работать с GitHub.
- CI/CD-системы, завязанные на webhook'и GitHub, не получали уведомлений о новых коммитах.

---

## Resolution & Recovery

1. Немедленно заблокирована запись в БД для защиты данных.
2. Выполнено восстановление основного кластера MySQL из резервной копии.
3. Восстановлена репликация между дата-центрами.
4. Проведена полная проверка целостности данных после восстановления.
5. Сервисы включались постепенно, с мониторингом нагрузки.

---

## Prevention & Action Items

1. **Улучшить устойчивость MySQL кластера** к сетевым сбоям (более безопасная обработка split-brain сценариев).
2. **Внедрить автоматическую защиту записи** при обнаружении аномалий в репликации.
3. **Улучшить инструменты оркестрации** для более быстрого восстановления в случае сбоев.
4. **Пересмотреть топологию БД** для снижения зависимости от одного дата-центра.
5. **Разработать систему оркестрации** для безопасного и быстрого отключения/включения сервисов.
6. **Улучшить процесс плановых работ с сетевым оборудованием** — более строгие проверки перед началом и более быстрое откатывание изменений.

---

## What Went Well

- Команда оперативно среагировала и предотвратила необратимую потерю данных.
- Платформа оставалась частично доступна (read-only).
- Коммуникация с пользователями через status.github.com была прозрачной и регулярной.
- Публикация подробного постмортема укрепила доверие сообщества.
